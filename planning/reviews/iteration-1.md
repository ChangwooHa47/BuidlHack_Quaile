# Iteration 1 Review

## Overall Assessment

**Not yet "개발만 하면 될 퀄리티."** The planning set is impressively thorough — ONE_PAGER / PRD / ERD / research notes are internally consistent on the big picture (secp256k1 ECDSA signing via `env::ecrecover`, Borsh canonicalization, Phase×Status state machine, pro-rata settlement). But there are several hard inconsistencies between ERD and the tasks (ContributionStatus enum set differs across files, `AttestationBundle` field set drifts, `live_start` present in ONE_PAGER Mermaid yet absent in the Rust struct, nonce replay storage key formula contradicts itself), and the cross-contract `contribute()` flow is specified in a way that cannot actually compile as described (one `#[payable]` method performing two view Promises and then mutating state — reentrancy/attached_deposit refund semantics are hand-waved). A seasoned NEAR engineer would reach these walls within the first hour of typing. Iteration 2 must close the ERD↔task drift and rewrite contract-03a's promise chain concretely before any other task is touched.

## Blockers (must fix before moving on)

- **ERD ↔ contract-02/tee-01 drift on `AttestationBundle`.** ERD §3.6 defines `signature_rs`, `signature_v`, `signing_key_id`, and `tee_report: Vec<u8>`. contract-02's spec drops `tee_report` from the contract struct (line 70–77) while tee-01 keeps it. Rust workspace will not compile two different definitions of the same struct. Decide: is `tee_report` part of the Borsh-serialized bundle, or attached alongside (and therefore NOT hashed / NOT part of `payload`)? Currently both interpretations co-exist.
- **`ContributionStatus` enum is inconsistent.** ERD §3.7 lists `Pending, Matched, Partial, Refunded, Claimed`. contract-03c then says "ERD update needed" to add `claim_done: bool` + `refund_done: bool` flags — which makes the enum insufficient for Partial, because a Partial contribution must be representable after claim-without-refund AND after refund-without-claim. The enum + flags design is under-specified: what is `status` after `claim()` on a Partial contribution? Still `Partial`? `Claimed` (with refund still pending)? ERD §5.2 state diagram does not match either option.
- **`SaleConfig` field set contradicts ONE_PAGER.** ONE_PAGER §5 and contract-01 both mention "Live" phase entry but ERD §3.2 `SaleConfig` has only `subscription_start / subscription_end / live_end` — no `live_start`. ERD §5.1 says `Subscribing → Live` transition triggers at `block_timestamp >= subscription_end`, so `live_start == subscription_end` is implied but never written. Decide explicitly; otherwise a dev will add the field "for clarity" and break Borsh.
- **`used_nonces` key definition collides across tasks.** ERD §8 says `used_nonces: UnorderedSet<(PolicyId, Nonce)>`. contract-03a says `used_nonces: LookupMap<[u8; 32], ()>` with `key = hash(policy_id || nonce)`. Two different storage types, two different keys. Borsh serialization of the tuple vs hashed key are not interchangeable — decide one.
- **contract-03a `contribute()` flow is not implementable as written.** The spec says "steps 1–16 run in a final callback." But `#[payable]` + `attached_deposit` is only accessible in the entry method, NOT in the callback (`env::attached_deposit()` in a callback returns 0). Also the spec says "실패 시 attached_deposit 자동 환불 (promise refund 패턴)" — NEAR has no automatic refund; you must explicitly return a `Promise::transfer(predecessor, deposit)` from the callback branch. Rewrite this flow step-by-step with actual function boundaries and state where `deposit` is stashed temporarily.
- **Cross-contract `is_eligible(bundle: AttestationBundle)` is a view call with a large struct argument.** A view call via Promise still consumes base gas and serializes the whole bundle (including `tee_report: Vec<u8>` if it is in the struct). This is feasible but the task does not estimate gas, and `tee_report` could be megabytes in a real attestation. Either (a) verify locally inside escrow (import verifier logic) or (b) pass only `payload_hash + signature` and store trust in verifier. Decide.
- **NEP-413 NEAR signature verification has no Python implementation path.** tee-03 says "ed25519 verify... NEP-413 follows"; it lists `nacl`/`cryptography` nowhere, and `py_near` may not expose verify. Verify which library provides `ed25519_verify` on raw bytes with the NEP-413 `sha256(borsh(…))` preimage before committing the dev to this.
- **tee-01 crate says `near_sdk::AccountId` lives behind `contract` feature, but the attestation Borsh layout depends on its serialization.** `AccountId` in near-sdk 5.x is `String`-backed; off-feature (pure std) you'd use `String` or `near_account_id::AccountId`. The Borsh wire format must match byte-for-byte. This is flagged as an Open Question but it is actually a blocker — the tee-05 Python hand-rolled Borsh uses `borsh_string(p.subject)`, which works only if Rust also serializes `AccountId` as `String`. Confirm and freeze.

## Major Issues

- **Settlement vs status-advance coupling.** ERD §5.1 says "Live → Closed is triggered when settle() completes (cross-contract call)." contract-03b settle() says "settle() 완료 시 PolicyRegistry.advance_status(Live→Closed)는 permissionless로 이미 호출 가능하니 같이 호출." But contract-01 says `mark_closed` is ONLY callable by the registered escrow account, and `advance_status` only transitions Upcoming→Subscribing→Live (not Live→Closed). So settle() must call `mark_closed`, not `advance_status`. Task contract-03b's "Open Questions §1" gets this wrong.
- **`advance_status` ordering bug.** contract-01 spec says `advance_status` is permissionless and advances "한 단계". The phase set is Upcoming → Subscribing → Live → Closed. But Live→Closed is gated by `mark_closed` (escrow-only). If `advance_status` is called while block_timestamp >= live_end but before settle(), what happens? The spec does not say. Either `advance_status` must refuse Live→Closed (and Live→Closed goes only through mark_closed), OR there is a race: settle() can be called on a `Closed` policy. Pin this.
- **Freshness windows are inconsistent.** PRD and ERD §3.3 say Persona `client_timestamp` must be within ±15 minutes. tee-02 says 15 min. tee-03 says 10 min (`freshness: now - proof.timestamp < 10 * 60 * 10**9`). Pick one and propagate.
- **Canonical `message` format mismatch.** ERD §3.3 says `"buidl-near-ai|{policy_id}|{nonce_hex}|{ts}"` for NEAR and `"…|{chain_id}|{address}"` for EVM. tee-03 defines a DIFFERENT format: `"buidl-near-ai|v1|{policy_id}|{nonce_hex}|{timestamp_ns}|{chain_descriptor}|{address}"` — adds `v1`, uses `eip155:{chain_id}` / `near:{network}` instead of raw chain_id. TEE signature verifier and client signer will disagree. Pick one SSOT (tee-03's looks better) and edit ERD.
- **`AttestationPayload.expires_at` vs nonce-binding.** PRD FR-AV-6 says "Attestation이 policy의 subscription 기간 내에 발급되었는지 검증한다" (Attestation issued within subscription window). contract-02 is_eligible checks only `expires_at > block_timestamp`. But replay protection via `used_nonces` lives in escrow (contract-03a), not verifier. A leftover Eligible bundle from policy A could in principle be ecrecover-valid against policy B if only policy_id in payload is what binds — and it is. OK, that works. But the spec does not explicitly state "payload.policy_id == contribute arg policy_id is the ONLY binding between bundle and target policy" — it is buried in contract-03a step 3. Worth making explicit in PRD §4.2/§4.3.
- **`signing_key_id` vs "just use the address".** contract-02 stores `signing_addresses: LookupMap<KeyId, [u8;20]>` and requires the bundle to carry `signing_key_id`. But the verifier ALREADY does `ecrecover(...) -> address`. Why carry a key_id at all? It saves O(1) lookup to avoid iterating, yes — but for MVP with 1–2 keys, a reverse `LookupMap<address, ()>` or a single `current + previous` pair is simpler. Either (a) drop `signing_key_id` from the bundle and do address-based lookup, or (b) keep it but ensure recovered-address equals `signing_addresses[signing_key_id]`. The current spec does the latter but the Python signer side never populates `signing_key_id` correctly (tee-05 has no `key_id` propagation to the payload, only `signing_key_id` on the bundle).
- **`scripts/deploy/config.env` uses `EthAddress` initialization as `[$BYTES]`.** This is a leftover placeholder. near-cli-rs takes JSON, and `[u8; 20]` in near-sdk init args is serialized as a JSON array of u8. Provide a helper: `python -c "import json,binascii; print(json.dumps(list(binascii.unhexlify('abcd...'))))"`. Otherwise a dev will guess and fail.
- **`settle()` offset is never stored.** contract-03b says "offset 저장 (다음 호출 위치)" but no field in `PolicyTotals` or `IdoEscrow` state holds this offset. Add `settle_cursor: LookupMap<PolicyId, u32>`.
- **`policy_pending_total` and `policy_investors` never cleaned up.** For MVP OK, but note it in state bloat section.
- **tee-02 `nonce format + uniqueness (via NEAR RPC view to ido-escrow.used_nonces)` is dangerous.** There is a TOCTOU race: TEE checks nonce unused → signs attestation → investor submits later → another investor races in. Since replay is actually checked at `contribute()` time (on-chain), TEE's nonce-uniqueness check is unnecessary and misleading. Delete it from the pipeline.
- **`env::keccak256_array` exists but only in recent near-sdk.** Verify near-sdk 5.3 actually exposes it (some versions only have `env::keccak256` returning `Vec<u8>`). contract-02's crypto.rs depends on this. If not, use `env::keccak256(...).try_into().unwrap()`.
- **eth_account `Account._sign_hash` is a private API.** tee-02 and tee-05 both call `Account._sign_hash(...)`. This has been renamed/removed in eth-account 0.13. Use `Account.signHash` or `eth_keys.keys.PrivateKey(pk).sign_msg_hash(h)`. Currently the spec will break on `pip install eth-account>=0.13`.
- **`Account._sign_hash(payload_hash.hex(), ...)` passes a hex string, not bytes.** tee-02 line 151 — wrong type. tee-05 passes bytes. Inconsistent.

## Security Concerns

- **TOCTOU on cross-contract view.** contribute() fetches `get_policy` + `is_eligible` via view Promises, then mutates state in callback. Between the views and the callback, block_timestamp advances, and `subscription_end` could elapse. A contribution accepted in the callback could be for a policy that is already past its window. Solution: re-read `block_timestamp()` in the final callback and re-check window. Not mentioned.
- **`is_eligible(bundle)` is a view on another contract but the bundle is a function argument.** A malicious caller can feed a bundle that passes verify but targets a different policy; contribute() step 3 checks `payload.policy_id == policy_id` locally so OK — but only if step 3 runs BEFORE the view. Flow order needs to be: local validation of bundle fields → cross-contract verify → state write. Document explicitly.
- **No `subject` binding on `is_eligible`.** Verifier's `is_eligible` does NOT check `payload.subject == predecessor_account_id`. That check lives in escrow (step 2). OK, but ensure verifier comment says "subject binding is caller's responsibility." contract-02 test case 9 mentions this; propagate to contract-02 Spec text too.
- **Refund-before-claim on `Partial`.** If a `Partial` contribution's `refund()` runs first and transfers the unmatched portion, then `claim()` later succeeds — fine. But if `refund()` runs and the `refund_done` flag toggles, a second `claim()` still works only if `claim_done` is a SEPARATE flag. Confirm the two flags are independent (task 03c says yes) but add an explicit test case: refund → claim order in both directions.
- **`attached_deposit` held across cross-contract callbacks.** If the final callback panics, `attached_deposit` is stranded in the escrow account with no Contribution record, i.e., a user can lose funds to a contract bug. Add a test and a defensive `Promise::new(predecessor).transfer(deposit)` on every failure branch.
- **Nonce uniqueness check only across `(policy_id, nonce)`.** If the TEE signs the same `(policy_id, nonce)` for a different subject, the escrow sees two different contributions with the same nonce — second is rejected. OK. But what about re-issuance across the grace period of key rotation? An attestation signed by key_1 during grace is replayable against the escrow even after rotation, because `used_nonces` isn't scoped by key_id. Low impact since nonce is still unique, but note it.
- **TEE signer privkey in env var.** PRD NFR-SEC-2 says "keys only generated in TEE or injected via attested channel." tee-02/05 punt to "MVP = env var." Document that this is explicitly not meeting NFR-SEC-2 and is a known MVP debt.
- **PII leakage via LLM rationale.** tee-04 has regex filters for ETH/NEAR addresses/email/URL. But LLM could emit GitHub login, project names, or self_intro fragments. The regex list is incomplete. At minimum, add a check: `rationale` must not contain any substring of `self_intro` or any `github.login` (before hashing). Also: consider banning digits > 4 consecutive (tx hashes, block numbers).
- **`self_intro[:2000]` sent to NEAR AI Cloud in judge call.** This is the "black box" path — the self_intro leaves our CVM and enters NEAR AI Cloud's CVM. PRD NFR-SEC-1 says "self_intro never leaves TEE." Contradicts research §5 decision "옵션 A: LLM 위탁." Resolve: either acknowledge NEAR AI Cloud IS part of the TEE boundary (needs remote attestation chain), or switch to on-CVM LLM for self_intro-touching calls.
- **GitHub OAuth token in Persona JSON in cleartext over TLS.** OK so long as the TEE endpoint terminates TLS inside CVM. Make that a bold NFR.

## Recommendations for Iteration 2

1. **Fix ERD first.** It is the SSOT but currently lies. Reconcile: ContributionStatus + claim_done/refund_done, AttestationBundle shape (tee_report in or out), canonical message format (v1 + chain_descriptor), used_nonces storage shape, SaleConfig `live_start` implicit-or-explicit, canonical AccountId Borsh encoding. Bump ERD to v2 and timestamp.
2. **Rewrite `contract-03a` contribute() flow in terms of NEAR Promise API primitives.** Name the entry method, the callback method, the state fields where deposit is staged, and the exact refund path. Include function signatures. Target: a NEAR dev could paste skeleton into `lib.rs` and fill in bodies.
3. **Settle vs mark_closed vs advance_status.** Rewrite the Live→Closed transition story in one place (ERD §5.1) and update contract-01 + contract-03b to match. Delete the `advance_status` Live→Closed path.
4. **Add missing tasks**: `contract-04-mock-ft`, `tee-06-key-bootstrap`, `test-02-cross-lang-borsh-vectors`.
5. **Resolve Open Questions 1, 2, 8 from research notes.** Especially #8 (TDX `report_data` encoding) — it affects `tee-05` sign_and_attach. Without this, the report binding may be wrong and the whole attestation report is effectively noise.
6. **Pin library versions and verify they exist.** Audit: `pysha3`, `py-near`, `eth-account._sign_hash`, `near-sdk 5.3` surface of `env::keccak256_array` and `env::ecrecover`. Replace broken ones. This should be a 30-minute task in itself.
7. **Add explicit gas budgets** to contract-03a (contribute: 2 view promises + 1 callback) and contract-03b (settle batch size). NEAR has ~300 TGas per transaction; know how many contributions `settle()` can process in one call.
8. **Decide the LLM trust boundary.** Option A (NEAR AI Cloud remote) requires nested attestation and conflicts with NFR-SEC-1 unless the outbound call is part of the TEE chain. Option B (on-CVM model) adds ops burden. Whatever you pick, rewrite `research §5~6` as the definitive decision with consequences enumerated, and update `tee-02`/`tee-04` accordingly.
9. **Add a per-task "who-writes-this-file" column** if you intend to run this via Dev Agent. Currently every task has Files but the ordering of which file to touch first (e.g., state.rs before subscription.rs) is implicit. For agent-driven dev, explicit sub-ordering matters.
10. **Re-run a third-party review after (1)–(8) are done.** Iteration 2's review should verify zero ERD drift and zero "Open Questions" on blocker-level decisions.
